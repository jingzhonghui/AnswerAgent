# 知识库与文件检索策略

AnswerAgent 不使用向量数据库（Chroma / Faiss / pgvector），
而是采用 **"关键词粗筛 + LLM 精读决策"** 的两阶段方案。

## 为什么不用 RAG 向量检索

- 向量库需要 Embedding 模型 + 持久化存储，引入较重依赖。
- 知识库文件更新需要重建索引，对个人/小团队使用场景不友好。
- 本项目知识库规模小（百量级文件），关键词命中已足够定位候选。

## 两阶段流程

### 阶段 1：关键词粗筛（本地，零 token）

1. 对用户问题做中英文混合分词：
   - 英文：正则 `[A-Za-z][A-Za-z0-9_]+`，转小写
   - 中文：`jieba.cut_for_search`
   - 过滤停用词
2. 扫描知识库内白名单后缀文件（`.md` `.py` `.ts` `.vue` 等）。
3. 评分加权：
   - 文件名命中: +3
   - 标题命中: +2
   - 路径命中: +1
   - 索引.md 提及该文件的段落命中: +2
4. 取 Top 10 作为候选。

### 阶段 2：LLM 精读（少量 token）

将候选文件的 `(rel_path, title)` 列表喂给 LLM，
让 LLM 选出 ≤ 5 个最相关文件，严格返回 JSON：

```json
{"files": ["overview.md", "architecture.md"]}
```

随后按知识库内相对路径读取完整内容（单文件 20KB 截断），
合并为问答链上下文。

## 知识库结构约定

```
backend/knowledge/{知识库名称}/
├── 索引.md           # 可选，存在时被 kb_router 和 file_loader 优先解析
├── *.md              # 知识文档
└── src/              # 可选，源码示例
    └── *.py / *.ts ...
```

知识库名称（一级目录名）允许中文，但禁止以 `.` 开头或包含路径分隔符。
