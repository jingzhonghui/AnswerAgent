# AnswerAgent 设计方案

## 1. 项目概述

AnswerAgent 是一个基于文件目录的问答智能体，支持从本地知识库中检索信息并结合 LLM 进行回答。采用 Python（FastAPI）后端 + Vue3 前端架构，通过 LangChain 框架统一管理 LLM 调用。

### 核心特性

- 基于文件目录的知识库管理，无需向量数据库
- 支持多知识库自动匹配与内容合并
- 支持 OpenAI 和 Anthropic 两种 LLM 提供商
- 多对话管理，支持新建、历史查看、重命名、删除
- SSE 流式输出，实时展示回答内容
- 亮色 / 暗色双主题

### 3.4 问答核心流程

```
用户提问
  │
  ▼
[Step 1] 知识库路由（kb_router.py）
  │  扫描 knowledge/ 获取所有知识库名称
  │  LLM 判断问题匹配哪些知识库（可多个）
  │
  ├─── 无匹配 ──────────────────────────────┐
  │                                         │
  ▼                                         │
[Step 2] 文件选取（file_loader.py）          │
  │  对每个匹配知识库并行执行：              │
  │                                         │
  │  阶段1：关键词粗筛（本地计算，零 token） │
  │    - 问题分词（jieba 支持中文）          │
  │    - 在文件名、md 标题、索引.md 中匹配   │
  │    - 若存在索引.md 优先解析其结构        │
  │    - 输出候选文件列表（最多 10 个）      │
  │                                         │
  │  阶段2：LLM 精读决策（少量 token）       │
  │    - 将候选文件摘要拼成简要目录          │
  │    - LLM 选出最相关文件（最多 5 个）     │
  │    - 读取完整内容 + 涉及的 src/ 源码     │
  │                                         │
  ▼                                         │
[Step 3] 合并上下文                          │
  │  将所有知识库的选定文件内容合并          │
  │                                         │
  ▼                                         ▼
[Step 4] LangChain 问答链（chain_builder.py）
  │  注入：合并上下文 / 对话历史（最近N轮） / 当前问题
  │  LLM 生成回答（SSE 流式输出）
  │
  ▼
前端实时展示
```

### 3.5 LangChain 问答链（LCEL）

```python
# 知识库问答链
kb_chain = (
    RunnablePassthrough.assign(
        context=lambda x: load_kb_context(x["question"], x["kb_names"]),
        history=lambda x: format_history(x["conversation_id"])
    )
    | kb_prompt
    | llm
    | StrOutputParser()
)

# 通用问答链（无知识库匹配时使用）
general_chain = (
    RunnablePassthrough.assign(
        history=lambda x: format_history(x["conversation_id"])
    )
    | general_prompt
    | llm
    | StrOutputParser()
)
```

**对话历史注入方式：**
- 取最近 `HISTORY_WINDOW` 轮（每轮 = 1 问 + 1 答）
- 以 `HumanMessage` / `AIMessage` 形式注入 `MessagesPlaceholder`

### 3.6 SSE 事件协议

流式问答接口 `POST /api/chat/stream` 返回以下 SSE 事件序列：

| 事件名 | 数据结构 | 说明 |
|--------|----------|------|
| `kb_matched` | `{"kb_names": ["newA", "newB"]}` | 匹配到的知识库列表，空数组表示通用回答 |
| `files_selected` | `{"kb": "newA", "files": ["api.md", "src/client.py"]}` | 每个知识库选定的文件（可多次） |
| `token` | `{"content": "根据"}` | 流式 token |
| `done` | `{"message_id": "uuid"}` | 回答完成 |
| `error` | `{"message": "错误信息"}` | 发生错误 |

### 4.4 流式输出交互细节

1. 发送问题后，立即显示"正在匹配知识库..."状态动画
2. 收到 `kb_matched` 事件后，在顶部显示知识库标签徽章
3. 收到 `files_selected` 事件后，在消息底部显示"参考文件"列表
4. 收到 `token` 事件后，逐字追加内容，末尾显示光标闪烁动画
5. 收到 `done` 事件后，光标消失，显示完整消息
6. 流式进行中，发送按钮变为"停止"按钮，点击后 abort SSE 连接

## 5. 关键设计决策说明

### 为什么不用 RAG（向量检索）？

RAG 方案需要向量数据库（Chroma、Faiss 等）和 Embedding 模型，引入了较重的依赖，且对知识库结构变更不友好（需要重建索引）。本项目采用"关键词粗筛 + LLM 精读决策"的两阶段文件选取方案：

- **粗筛阶段**：纯本地计算，零 token 消耗，快速缩小候选范围
- **精读阶段**：仅传递文件摘要（非全文），token 消耗可控
- 知识库文件修改后即时生效，无需重建索引

### 为什么选择 SSE 而非 WebSocket？

SSE 是单向推送（服务器→客户端），适合流式输出场景，实现更简单，原生浏览器支持无需额外库。WebSocket 适合双向实时通信场景，本项目不需要。

### 为什么对话存为独立 JSON 文件而非数据库？

- 零依赖，无需安装配置数据库
- 每个对话独立文件，便于查看、备份、迁移
- 对话数量在个人使用场景下不会到达文件系统性能瓶颈
