# AnswerAgent 开发任务拆分计划

## Context

当前仓库处于“设计文档 + HTML 原型”阶段，实际还没有 `backend/` 或 `frontend/` 工程。`docs/设计方案.md` 已定义目标架构：Python FastAPI 后端、Vue3 TypeScript 前端、本地目录知识库、JSON 对话持久化、LangChain 默认 LCEL 问答链、SSE 流式输出，以及 Deep Agents 的可选复杂任务模式。

本计划的目标是把设计方案拆成可执行开发任务，优先形成可运行的 MVP 闭环，再在默认链路稳定后接入 Deep Agents。默认主流程保持轻量可控：`kb_router.py → file_loader.py → chain_builder.py → SSE`，Deep Agents 仅作为 `mode=agent` 的增强路径。

## 总体开发顺序

1. 工程骨架与协议基线
2. 对话 JSON 持久化与 CRUD
3. 知识库扫描、路由和文件选择
4. 默认 LCEL 问答链与后端 SSE
5. Vue3 前端工程化迁移原型
6. 前端接入 POST SSE 流式协议
7. 端到端联调与 README/示例知识库
8. Deep Agents 可选模式

## 关键技术决策

- 保持 `POST /api/chat/stream`，因为发送消息会创建/更新对话，语义上有副作用。
- 前端不要使用原生 `EventSource`，因为它只支持 GET，无法自然发送 JSON body。
- 推荐前端使用 `@microsoft/fetch-event-source` 或等价 `fetch + ReadableStream` 封装来消费 POST SSE；第一版优先使用 `@microsoft/fetch-event-source` 降低解析复杂度。
- Deep Agents 不进入 MVP 主链路，默认 `mode=default` 必须先稳定。
- 所有知识库文件读取必须限制在 `KNOWLEDGE_PATH` 内，禁止路径穿越。

## 里程碑 1：对话 JSON 持久化与 CRUD

### 目标

在不接 LLM 的情况下，先完成真实对话生命周期。

### 关键文件

- `backend/app/core/chat_manager.py`
- `backend/app/api/conversations.py`
- `backend/app/models/schemas.py`
- `frontend/src/api/index.ts`
- `frontend/src/stores/chat.ts`
- `frontend/src/components/ConversationSidebar.vue`
- `frontend/src/components/ChatWindow.vue`

### 任务

- 实现 `chat_manager.py`：创建、列表、详情、重命名、删除、追加 user/assistant 消息、读取最近历史。
- 每个对话保存为 `backend/data/conversations/{id}.json`。
- 实现对话 API：
  - `GET /api/conversations`
  - `POST /api/conversations`
  - `GET /api/conversations/{id}`
  - `DELETE /api/conversations/{id}`
  - `PATCH /api/conversations/{id}/title`
- 前端接入对话 API，替换原型中的 localStorage mock 数据。
- 实现对话列表、新建、选择、重命名、删除。

### 验收标准

- 新建对话会生成 JSON 文件。
- 刷新页面后对话列表仍存在。
- 重命名和删除会同步更新文件系统。
- 删除当前对话后前端能切换到剩余对话。
- 不允许通过路径穿越访问非对话 JSON 文件。

## 里程碑 2：知识库扫描、路由和文件选择

### 目标

落地“本地目录知识库，无向量数据库”的检索基础。

### 关键文件

- `backend/app/api/knowledge_bases.py`
- `backend/app/core/kb_router.py`
- `backend/app/core/file_loader.py`
- `backend/app/core/llm_factory.py`
- `backend/knowledge/demo/索引.md`
- `backend/knowledge/demo/overview.md`

### 任务

- 实现 `GET /api/knowledge-bases`，扫描 `KNOWLEDGE_PATH` 下一级目录。
- 建立 demo 知识库，包含 `索引.md`、Markdown 文档和可选源码文件。
- 实现 `kb_router.py`：
  - 第一版支持本地启发式匹配。
  - 配置 LLM 后可用 LLM 从候选知识库中多选。
  - 无 API key 时保留本地匹配能力，便于开发验收。
- 实现 `file_loader.py`：
  - 扫描 Markdown 和常见源码文本文件。
  - 解析文件名、路径、Markdown 标题、`索引.md`。
  - 使用 `jieba` 和英文 token 做关键词粗筛。
  - 候选最多 10 个，精选最多 5 个。
  - 控制单文件和总上下文字符上限。
  - 返回知识库内相对路径，不暴露绝对路径。
- 所有文件读取必须 `resolve` 后校验仍位于知识库根目录内。

### 验收标准

- demo 知识库可被接口列出。
- 给定问题能匹配出知识库和候选文件。
- 知识库为空时返回空列表，不导致服务崩溃。
- 路径穿越尝试必须失败。
- 大文件不会无限制进入上下文。

## 里程碑 3：默认 LCEL 问答链与后端 SSE

### 目标

完成默认问答链：匹配知识库、选择文件、调用 LLM、SSE 流式输出并持久化最终消息。

### 关键文件

- `backend/app/api/chat.py`
- `backend/app/core/chain_builder.py`
- `backend/app/core/llm_factory.py`
- `backend/app/core/kb_router.py`
- `backend/app/core/file_loader.py`
- `backend/app/core/chat_manager.py`

### 任务

- 实现 `llm_factory.py`，统一创建 OpenAI / Anthropic LangChain chat model。
- 实现 `chain_builder.py`：
  - 知识库问答链：注入上下文、最近 `HISTORY_WINDOW` 轮历史、当前问题。
  - 通用问答链：无知识库命中时只注入历史和问题。
- 实现 `POST /api/chat/stream` 的 `mode=default`：
  - 校验或创建对话。
  - 先保存 user message。
  - 发出 `kb_matched`。
  - 发出一个或多个 `files_selected`。
  - 流式发出 `token`。
  - 完成后保存 assistant message。
  - 发出 `done`，建议包含 `conversation_id` 和 `message_id`。
  - 异常时发出 `error`。
- 若收到 `mode=agent`，第一版返回未实现错误，不影响 default。
- 处理客户端断开和停止生成。

### 验收标准

- 使用 curl/API 客户端可看到 `kb_matched → files_selected → token* → done`。
- 命中知识库时保存 `kb_names` 和 `files_used`。
- 未命中时 `kb_matched.kb_names` 是空数组，并走通用回答。
- 回答完成后重新读取对话详情能看到 user 和 assistant 消息。
- LLM 配置错误能通过 `error` 事件返回，不让前端无限等待。

## 里程碑 5：前端接入 POST SSE 流式协议

### 目标

完成真实浏览器端流式问答体验。

### 关键文件

- `frontend/package.json`
- `frontend/src/api/index.ts`
- `frontend/src/stores/chat.ts`
- `frontend/src/components/InputBar.vue`
- `frontend/src/components/MessageBubble.vue`
- `backend/app/api/chat.py`

### 任务

- 添加 `@microsoft/fetch-event-source` 依赖，或实现等价 fetch SSE parser。
- `api/index.ts` 封装 `streamChat(request, handlers, signal)`。
- `sendMessage`：立即追加 user message 和 assistant draft。
- 处理 SSE 事件：
  - `kb_matched`：更新匹配状态。
  - `files_selected`：更新 draft 的参考文件。
  - `token`：增量追加回答内容。
  - `done`：结束 streaming，并重新拉取对话详情校准。
  - `error`：结束 streaming 并展示错误。
- `stopStreaming` 使用 `AbortController.abort()`。
- 后端确保 SSE 响应头正确，`data` 始终为 JSON 字符串。

### 验收标准

- 浏览器中发送问题后能实时追加 token。
- 参考文件在收到 `files_selected` 后出现。
- 点击停止能中断前端请求，并清理生成状态。
- 错误事件不会让 UI 卡在生成中。
- `done` 后刷新页面，回答从后端 JSON 恢复。

## 里程碑 7：Deep Agents 可选模式

### 目标

在默认链路稳定后，接入 `mode=agent` 复杂任务模式。

### 关键文件

- `backend/requirements.txt`
- `backend/app/core/agent_builder.py`
- `backend/app/core/agent_tools.py`
- `backend/app/core/agent_stream_adapter.py`
- `backend/app/api/chat.py`
- `backend/app/models/schemas.py`
- `frontend/src/stores/chat.ts`
- `frontend/src/components/InputBar.vue`

### 任务

- 在 `requirements.txt` 增加 `deepagents`。
- `agent_tools.py` 只封装受控知识库工具：
  - `list_knowledge_bases`
  - `route_knowledge_bases`
  - `search_kb_files`
  - `select_kb_files`
  - `read_kb_files`
- 工具底层复用 `kb_router.py` 和 `file_loader.py`。
- 禁止开放通用文件读写、shell execute、`.env` 读取、`data/conversations` 直接访问、`knowledge` 写入。
- `agent_builder.py` 创建 Deep Agent，注入受控 tools 和严格 system prompt。
- `agent_stream_adapter.py` 将 Deep Agents 事件映射为现有 SSE：
  - 工具路由结果 → `kb_matched`
  - 文件选择结果 → `files_selected`
  - 主回答增量 → `token`
  - 完成 → `done`
  - 异常 → `error`
- assistant 消息保存可选字段：`agent_mode`、`agent_trace_summary`。
- 前端增加模式选择：默认问答 / 复杂任务。

### 验收标准

- `mode=default` 行为不变。
- `mode=agent` 能处理复杂问题，并仍返回同一套 SSE 事件。
- Agent 引用文件均来自 `KNOWLEDGE_PATH` 下的相对路径。
- Deep Agents 配置错误不会影响默认模式。
- 对话 JSON 能区分 `agent_mode`。

## 风险与控制

| 风险 | 控制方式 |
|---|---|
| POST SSE 与 EventSource 不匹配 | 使用 `@microsoft/fetch-event-source`，不使用原生 EventSource |
| LLM JSON 输出不稳定 | 本地启发式兜底，LLM 结果做容错解析 |
| 大文件导致上下文超限 | 单文件和总上下文字符上限 |
| 路径穿越读取敏感文件 | 所有路径 resolve 后校验在知识库根目录内 |
| JSON 并发写入覆盖 | 第一版使用原子替换，必要时加简单文件锁 |
| 前端流式状态卡住 | `done/error/abort` 统一清理状态 |
| Deep Agents 过早引入导致 MVP 失焦 | 放在默认链路和 SSE 稳定之后 |

- 验证对话新建、选择、重命名、删除。

- 点击停止，确认请求中断且 UI 状态清理。

### 端到端

- 按 README 从空数据目录启动。
- 新建对话并发送命中 demo 知识库的问题。
- 确认 `kb_matched`、`files_selected`、`token`、`done` 全部可见或可从 Network 观察。
- 刷新页面后确认对话和消息从 JSON 恢复。
- 删除对话后确认 JSON 文件删除。

## MVP 范围

MVP 包含：工程骨架、对话 CRUD、本地知识库扫描、默认 LLM 问答、POST SSE、前端流式展示、停止生成、参考文件、JSON 持久化、亮暗主题、README 和 demo 知识库。

MVP 暂不包含：Deep Agents、多用户、登录鉴权、向量库、文件上传、知识库在线编辑、自动重连、agent trace 可视化。Deep Agents 作为后续增强单独实施。
